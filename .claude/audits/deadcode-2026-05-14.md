Findings below are exported functions with **zero call sites** in the
codebase. They split into three buckets: (A) auto-removed in this pass,
(B) wire-up gaps where the function exists but the consumer that was
supposed to call it was never added, (C) interface-satisfaction false
positives that must stay.

## C. False positives (interface conformance)

`deadcode` flags these because nothing calls them through the concrete
type. They are required by the `_ provider.ContainerProvider = …` style
canary assertions in `internal/provider/types_test.go` (lines 78–84 and
similar). Removing them breaks the compile-time interface check.

- `mockContainerProvider.{StopCrewRuntime,RemoveCrewRuntime,ContainerStats,Exec,ExecInspect,CopyToContainer,RemoveCrewVolumes,ExecInteractive,ExecResize}`
- `mockStateProvider.List`
- `mockStorageProvider.{List,ListRecursive,EnsureDir,Watch}`
- `noopReader.Read` (backup collector — same pattern)
- `journalCaptureEmitter.Flush` (telemetry test emitter — must satisfy emitter interface)

**Do not remove.** If `deadcode` runs in CI, add `//deadcode:ignore` comments
or an exclude list scoped to these symbols.

## Appendix — `govulncheck` snapshot (2026-05-15)

`govulncheck ./...` flagged 2 advisories, both in `github.com/docker/docker@v28.5.2+incompatible`:

| ID | Title | Status |
|---|---|---|
| GO-2026-4887 | AuthZ plugin bypass when provided oversized request bodies | Fixed in: N/A |
| GO-2026-4883 | Off-by-one error in plugin privilege validation | Fixed in: N/A |

Both are **daemon-side** issues that require Docker AuthZ plugins to be
loaded on the host running dockerd. Crewship is the *client* of the
Docker daemon, not the daemon itself — these code paths are reachable
only on the operator's host, and only if they've explicitly installed
a third-party AuthZ plugin. The vuln set is informational for our
threat model, not a Crewship fix.

No upstream fix exists yet (`Fixed in: N/A`); re-check after the next
moby release.
