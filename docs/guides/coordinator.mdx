# Coordinator

The Coordinator is a workspace-level agent role in Crewship. While a **Lead** manages a single crew, the **Coordinator** sees every crew and every agent across the entire workspace. Its job is to decompose complex, multi-crew objectives into targeted missions, submit proposals for human approval, and monitor progress across all teams.

<Note>
  The Coordinator role is sometimes referred to internally as "CEO" in system prompt context. Despite this label, the Coordinator holds no hierarchical authority over any other agent. It is a functional distinction, not a rank.
</Note>

## Coordinator vs Lead

| Aspect | Lead | Coordinator |
|--------|------|-------------|
| **Scope** | Single crew | Entire workspace (all crews) |
| **Visibility** | Own crew members and tasks | All crews, all agents, all missions |
| **Mission creation** | Direct -- creates and starts missions in own crew | Proposals -- submits for human review, or direct for urgent tasks |
| **System prompt block** | `[CREW CONTEXT]` with own members | `[COORDINATOR CONTEXT]` with all crews and active missions |
| **Typical actions** | Assign tasks, query agents, manage crew | Decompose objectives, create crews, connect crews, monitor workspace |

## The Equality Principle

<Warning>
  The Coordinator is **not above anyone**. This is a foundational, non-overridable principle of the Crewship architecture.
</Warning>

From the Crewship ethos injected into every Coordinator's system prompt:

> You are a workspace member with coordination responsibility on the Crewship --
> connecting the expeditions of all crews towards one shared goal. You are not above
> anyone -- you are an equal who sees the bigger picture and helps crews align
> their efforts towards the common adventure.

The hierarchy in Crewship is **functional** (who orchestrates what), not **value-based** (who matters more). The Coordinator carries a wider lens, but a Lead's domain expertise within their crew is equally vital. This equality creates trust -- agents collaborate freely because they know the Coordinator is one of them, just with a different task.

## Coordinator System Prompt Context

When a Coordinator agent is invoked, it receives a `[COORDINATOR CONTEXT]` block in its system prompt containing:

1. **All workspace crews** with their agents (name, slug, role title, description, crew ID)
2. **Active missions** across all crews (status, title, crew slug, mission ID)
3. **Proposal workflow** instructions with curl examples
4. **Direct mission** creation instructions for urgent tasks
5. **Crew management** commands (create crew, create agent)
6. **Monitoring commands** (list connections, check status, list templates)

This context is built by `BuildCoordinatorContext()` in `internal/orchestrator/lead.go` and refreshed each time the Coordinator is invoked.

## Sidecar API Endpoints

The Coordinator accesses all its capabilities through the sidecar HTTP API running at `http://localhost:9119` inside the container. Below is every endpoint available to a Coordinator agent.

### Workspace Discovery

<Accordion title="GET /crews -- List all workspace crews">
  Returns every crew in the workspace with their metadata.

  ```bash
  curl -s http://localhost:9119/crews
  ```

  **Response:** Array of crew objects with `id`, `name`, `slug`, `description`, and nested agent lists.

  Use this to discover available crews before creating missions or proposals.
</Accordion>

<Accordion title="GET /crew-connections -- List crew connection topology">
  Returns the directed graph of crew-to-crew connections in the workspace.

  ```bash
  curl -s http://localhost:9119/crew-connections
  ```

  **Response:** Array of connection objects showing which crews can communicate with each other.

  Connections enable cross-crew messaging and file sharing between crews.
</Accordion>

<Accordion title="GET /credentials -- List workspace credentials">
  Returns all credentials registered in the workspace (names and types, not secrets).

  ```bash
  curl -s http://localhost:9119/credentials
  ```

  **Response:** Array of credential objects with `id`, `name`, `type`, and assignment info.

  Use this to discover what credentials are available before assigning them to newly created agents.
</Accordion>

### Crew Management

<Accordion title="POST /crew/create -- Create a new crew">
  Creates a new crew in the workspace.

  ```bash
  curl -s -X POST http://localhost:9119/crew/create \
    -H "Content-Type: application/json" \
    -d '{
      "name": "Data Pipeline",
      "slug": "data-pipeline",
      "description": "ETL and data processing crew",
      "icon": "database",
      "color": "cyan"
    }'
  ```

  **Response:**
  ```json
  {
    "id": "<crew_id>",
    "name": "Data Pipeline",
    "slug": "data-pipeline",
    "workspace_id": "..."
  }
  ```

  <Tip>
    Use lucide icon names for `icon` (e.g., `code`, `rocket`, `clipboard`) and palette IDs for `color` (e.g., `blue`, `emerald`, `violet`, `amber`, `rose`, `cyan`, `lime`, `fuchsia`).
  </Tip>
</Accordion>

<Accordion title="POST /agent/create -- Create a new agent">
  Adds an agent to an existing crew.

  ```bash
  curl -s -X POST http://localhost:9119/agent/create \
    -H "Content-Type: application/json" \
    -d '{
      "crew_id": "<crew_id>",
      "name": "Alice",
      "role_title": "Lead Developer",
      "agent_role": "LEAD",
      "description": "Manages the dev crew and reviews PRs",
      "system_prompt": "You are a senior developer...",
      "cli_adapter": "CLAUDE_CODE",
      "tool_profile": "CODING"
    }'
  ```

  **Parameters:**
  | Field | Required | Options |
  |-------|----------|---------|
  | `crew_id` | Yes | ID of the target crew |
  | `name` | Yes | Display name |
  | `agent_role` | Yes | `LEAD`, `AGENT`, or `COORDINATOR` |
  | `cli_adapter` | Yes | `CLAUDE_CODE`, `CODEX_CLI`, `GEMINI_CLI`, `OPENCODE` |
  | `tool_profile` | Yes | `CODING`, `MINIMAL`, `FULL` |

  **Response:**
  ```json
  {
    "id": "<agent_id>",
    "name": "Alice",
    "slug": "alice",
    "crew_id": "<crew_id>"
  }
  ```
</Accordion>

<Accordion title="POST /agent-credentials -- Assign credential to agent">
  Assigns an existing workspace credential to an agent.

  ```bash
  curl -s -X POST http://localhost:9119/agent-credentials \
    -H "Content-Type: application/json" \
    -d '{
      "agent_id": "<agent_id>",
      "credential_id": "<credential_id>"
    }'
  ```

  Use `GET /credentials` first to discover available credential IDs.
</Accordion>

<Accordion title="POST /crew-connections -- Create crew connection">
  Establishes a directed connection between two crews, enabling cross-crew messaging and file sharing.

  ```bash
  curl -s -X POST http://localhost:9119/crew-connections \
    -H "Content-Type: application/json" \
    -d '{
      "from_crew_id": "<crew_a_id>",
      "to_crew_id": "<crew_b_id>"
    }'
  ```

  Connections are directional. Create two connections (A to B and B to A) for bidirectional communication.
</Accordion>

### Mission Proposals

The proposal workflow is the **recommended path** for multi-crew objectives. The Coordinator submits a structured plan, a human reviews it, and on approval the system automatically creates all missions and tasks.

<Accordion title="POST /proposal -- Submit mission proposal">
  Submits a proposal containing one or more missions for human review.

  ```bash
  curl -s -X POST http://localhost:9119/proposal \
    -H "Content-Type: application/json" \
    -d '{
      "title": "Q2 Platform Migration",
      "description": "Migrate frontend and backend to new infra",
      "plan": "Phase 1: Backend API migration. Phase 2: Frontend rebuild.",
      "missions": [
        {
          "crew_id": "<backend_crew_id>",
          "title": "Migrate API to new cluster",
          "tasks": [
            {
              "title": "Update deployment configs",
              "assigned_agent_id": "<agent_id>",
              "task_order": 1
            },
            {
              "title": "Run integration tests",
              "assigned_agent_id": "<agent_id>",
              "task_order": 2,
              "depends_on": ["Update deployment configs"]
            }
          ]
        },
        {
          "crew_id": "<frontend_crew_id>",
          "title": "Rebuild frontend components",
          "tasks": [
            {
              "title": "Migrate to new API endpoints",
              "assigned_agent_id": "<agent_id>",
              "task_order": 1
            }
          ]
        }
      ]
    }'
  ```

  **Response:**
  ```json
  {
    "id": "<proposal_id>",
    "status": "PENDING"
  }
  ```

  Each mission in the proposal targets a specific crew (by `crew_id`) and contains tasks with agent assignments and ordering.
</Accordion>

<Accordion title="GET /proposals -- List pending proposals">
  Returns all proposals in the workspace, ordered by creation date (newest first).

  ```bash
  curl -s http://localhost:9119/proposals
  ```

  **Response:** Array of proposal objects with `id`, `title`, `status` (`PENDING`, `APPROVED`, `REJECTED`), `missions`, `reviewed_by`, `review_notes`, and timestamps.

  Use this to check whether your submitted proposals have been reviewed.
</Accordion>

### Proposal Lifecycle

<Steps>
  <Step title="Coordinator submits proposal">
    The Coordinator analyzes a complex objective, decomposes it into crew-specific missions with tasks, and calls `POST /proposal`.
  </Step>
  <Step title="Proposal enters PENDING state">
    The proposal is stored with `status: PENDING` and a real-time WebSocket event (`proposal_created`) notifies the UI.
  </Step>
  <Step title="Human reviews in the UI">
    A human operator sees the proposal in the Crewship dashboard. They can inspect the plan, the targeted crews, task assignments, and dependencies.
  </Step>
  <Step title="Approve or Reject">
    - **Approve:** The system atomically creates all missions and their tasks in the database. Each mission is linked to its crew's Lead agent. Tasks with dependencies start in `BLOCKED` status; others start as `PENDING`. The response includes the created `mission_ids`.
    - **Reject:** The proposal is marked `REJECTED` with optional `review_notes` explaining why. No missions are created.
  </Step>
  <Step title="Missions execute automatically">
    Once approved, the MissionEngine picks up the new missions. Lead agents in each crew receive their missions and begin task execution through the standard orchestration loop.
  </Step>
</Steps>

### Direct Mission Creation

For urgent tasks that should not wait for human review, the Coordinator can create missions directly.

<Accordion title="POST /mission/create -- Create mission directly">
  Creates a mission immediately, bypassing the proposal workflow.

  ```bash
  curl -s -X POST http://localhost:9119/mission/create \
    -H "Content-Type: application/json" \
    -d '{
      "title": "Hotfix: API rate limit",
      "crew_id": "<target_crew_id>",
      "tasks": [
        {
          "title": "Fix rate limiter config",
          "assigned_to_id": "<agent_id>",
          "task_order": 1
        }
      ]
    }'
  ```

  <Warning>
    Coordinators **must** include `crew_id` explicitly. Unlike Lead agents (who default to their own crew), the Coordinator has no implicit crew scope.
  </Warning>

  After creation, start the mission:
  ```bash
  curl -s -X POST http://localhost:9119/mission/<mission_id>/start
  ```
</Accordion>

</Accordion>

<Accordion title="GET /missions/all/summary -- Aggregated mission summary">

  **Response:**

<Accordion title="GET /mission/{id} -- Single mission status">

</Accordion>

### Communication and Escalation

<Accordion title="POST /escalate -- Escalate to human">
  Flags an issue for human attention. The call **blocks** until the human responds (up to 5 minutes) or times out.

  ```bash
  curl -s -X POST http://localhost:9119/escalate \
    -H "Content-Type: application/json" \
    -d '{
      "from": "coordinator-agent",
      "reason": "Two crews have conflicting deployment schedules",
      "context": "Backend crew plans to deploy at 14:00 but frontend needs stable API",
      "type": "CONFLICT"
    }'
  ```

  **Response (human responded):**
  ```json
  {
    "escalation_id": "<id>",
    "status": "RESOLVED",
    "resolution": "Delay backend deploy to 16:00"
  }
  ```

  **Response (timeout):**
  ```json
  {
    "escalation_id": "<id>",
    "status": "TIMEOUT",
    "resolution": ""
  }
  ```

  The escalation is created first (short timeout), then the sidecar long-polls for up to 300 seconds waiting for the human response.
</Accordion>

<Accordion title="POST /report-confidence -- Report task confidence">
  Reports the agent's mid-task confidence level to the system. Used for monitoring and automated intervention.

  ```bash
  curl -s -X POST http://localhost:9119/report-confidence \
    -H "Content-Type: application/json" \
    -d '{
      "confidence": 0.4,
      "reason": "Unfamiliar with the target framework, may need guidance"
    }'
  ```

  The `confidence` value is a float between 0.0 and 1.0. Low confidence reports may trigger lead intervention or escalation.
</Accordion>

<Accordion title="POST /query -- Query a crew member">
  Asks a peer agent a synchronous question. The caller blocks until the target responds (up to 120 seconds).

  ```bash
  curl -s -X POST http://localhost:9119/query \
    -H "Content-Type: application/json" \
    -d '{
      "target": "alice",
      "question": "What is the current API schema version?",
      "from": "coordinator-agent"
    }'
  ```

  Queries have a **depth limit of 2** to prevent infinite loops (agent A queries B, B queries C, but C cannot query further).
</Accordion>

### Manifest Management

<Accordion title="GET /manifest -- Read crew manifest">
  Returns the current crew container manifest, which tracks installed packages, credentials, and setup commands.

  ```bash
  curl -s http://localhost:9119/manifest
  ```

  **Response:**
  ```json
  {
    "version": 1,
    "packages": {
      "apt": ["git", "curl"],
      "npm": ["typescript"],
      "pip": []
    },
    "credentials": [
      {"name": "github-token", "agent": "alice", "type": "BEARER"}
    ],
    "setup_commands": ["git config --global user.name Crewship"]
  }
  ```
</Accordion>

<Accordion title="PATCH /manifest -- Update crew manifest">
  Merges additions into the existing manifest. Duplicate entries are deduplicated automatically.

  ```bash
  curl -s -X PATCH http://localhost:9119/manifest \
    -H "Content-Type: application/json" \
    -d '{
      "packages": {"apt": ["jq"]},
      "setup_commands": ["echo setup complete"]
    }'
  ```

  The manifest is written atomically (temp file + rename) to prevent partial reads.
</Accordion>

## Cross-Crew Orchestration Patterns

### Pattern 1: Coordinated Multi-Crew Deployment

The Coordinator decomposes a deployment into crew-specific missions with ordering:

1. Submit a proposal with missions for the backend crew (database migration), platform crew (infrastructure), and frontend crew (deploy new build).
2. Use `depends_on` within each mission to enforce task ordering.
3. Monitor with `GET /missions/all/summary` and drill into individual missions as needed.
4. If conflicts arise between crews, use `POST /escalate` to involve a human.

### Pattern 2: Dynamic Crew Provisioning

When the Coordinator identifies a gap in workspace capabilities:

1. `POST /crew/create` to create a new specialized crew.
2. `POST /agent/create` to populate it with agents (Lead + specialized agents).
3. `GET /credentials` followed by `POST /agent-credentials` to equip agents.
4. `POST /crew-connections` to connect the new crew to relevant existing crews.
5. Submit a proposal targeting the new crew.

### Pattern 3: Workspace Health Check

A periodic monitoring workflow:

1. `GET /missions/all/summary` for high-level status counts.
2. `GET /missions/all` to identify stalled or failed missions.
3. `GET /crews` to check agent availability.
4. `GET /crew-connections` to verify topology is complete.
5. `POST /escalate` if systemic issues are detected.

<Tip>
  The Coordinator should prefer the **proposal workflow** over direct mission creation for any non-urgent, multi-crew objective. Proposals create an audit trail and give humans the opportunity to refine the plan before execution begins.
</Tip>
