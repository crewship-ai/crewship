## B. ARCHITEKTURNI OTAZKY (dablovy advokat)

### B1. Go service pristupuje k DB — nebo ne?

- `architecture.md`: "Go service does NOT do: No database access (Next.js owns PostgreSQL via Prisma)"
- `DEPLOYMENT.md`: `DATABASE_URL` je env var pro crewshipd (Go)
- `API.md`: "crewshipd validates X-Webhook-Secret (IPC → Next.js → DB)"

**Problem:** Webhook validace pres IPC je POMALA. Kazdy webhook = HTTP request na Unix socket → Next.js → Prisma → PostgreSQL → response. Pro 60 webhooks/min to muze byt bottleneck.

**Otazka:** Ma crewshipd cist primo z PostgreSQL (jen SELECT, bez Prisma)?
- **PRO:** Rychlejsi webhook validace, agent status, session metadata
- **PROTI:** Dva pristupy k DB = konzistence problem, dva connection pooly

**Muj nazor:** Pro MVP nech IPC. Pokud bude bottleneck, pridat read-only PostgreSQL pristup do Go v Phase 2. Unix socket HTTP je < 1ms latence — staci.

### B2. Kdo vytvari ConversationSession radek v DB?

Flow:
1. Uzivatel klikne "New chat" → Next.js REST API vytvori ConversationSession v PostgreSQL
2. Next.js posle pres IPC na crewshipd "start agent, session_id=X"
3. crewshipd spusti Docker exec, zacne psat do JSONL

**Otazka:** Co kdyz webhook spusti agenta? Kdo vytvori session?
- Webhook prijde do crewshipd → crewshipd musi vytvorit session... ale nema pristup k DB!
- Takze: crewshipd posle IPC request na Next.js "create session for agent X, trigger=webhook"
- Next.js vytvori session a vrati session_id
- crewshipd spusti agenta

**Problem:** Kruhova IPC — webhook prijde do Go, Go musi pozadat Next.js o DB operaci, Next.js odpovi, Go pokracuje. Funguje, ale je to dalsi roundtrip.

**Reseni:** Prijatelne pro MVP. Go posle jeden IPC request "validate webhook + create session" a Next.js udela oboje v jednom handeru.

### B3. JSONL konverzace — jak hleda uzivatel?

Uzivatel chce: "Ukaž mi všechny chaty kde agent zmínil slovo 'error'"

S PostgreSQL: `WHERE content LIKE '%error%'` — jednoduche.
S JSONL: crewshipd musi otevrit KAZDY JSONL soubor a grepovat.

**Otazka:** Je search across sessions P0 feature?
- PRD.md CHAT-03 rika "Historie konverzaci" — ale to je seznam sessions (metadata v PostgreSQL), ne fulltext search.
- Search uvnitr jedne session = cteni jednoho JSONL souboru = OK.
- Search across sessions = problem.

**Muj nazor:** Pro MVP staci search uvnitr session (Go precte jeden JSONL). Cross-session search je Phase 2 (SQLite FTS, MeiliSearch, nebo zpetna indexace do PostgreSQL).

### B4. Kolik RAM ma Mac Mini 16GB pro agenty?

```
Next.js:     ~300 MB
crewshipd:   ~50 MB
PostgreSQL:  ~200 MB
Docker:      ~200 MB overhead
---
Volne:       ~15.2 GB

1 agent kontejner (4GB default): 15.2 / 4 = 3 kontejnery max
```

**Problem:** Default 4GB per team container je MOC pro 16GB dev stroj. S 3 tymy je plno.

**Reseni:** Pro dev snizit default na 1GB (`container_memory_mb` v Team modelu). 4GB je pro produkci.

### B5. Agent CLI tool startup time

Claude Code cold start: ~2-5s (Node.js, npm globals).
Pro kazdy Docker exec = novy proces = novy cold start.

**Otazka:** Pokud uzivatel posle 10 zprav, je to 10x cold start?

**Odpoved:** Ano, v soucasnem navrhu kazda zprava = novy `docker exec` = novy CLI process. To je pomale.

**Alternativy:**
1. **Persistent CLI session** (OpenCode serve, Codex JSON-RPC) — CLI bezi nepretrzite, zpravy se posilaji pres API. NEJLEPSI pro UX, ale slozitejsi na implementaci.
2. **Batch mode** — posli historii konverzace pri kazdem exec. Jednodussi, ale pomale pro dlouhe konverzace.
3. **Hybrid** — prvni zprava = cold start, dalsi zpravy = persistent session (pokud CLI podporuje).

**Muj nazor:** PRD.md (RUN-04) rika "CLI session management" — to implicitne rika persistent session. Pro MVP: pouzit `docker exec` s pipe (stdin) pro continuous session pokud CLI podporuje, jinak batch mode.

### B6. Co kdyz crewshipd crashne uprostred agent runu?

1. crewshipd crashne
2. Docker exec procesess STALE bezi v kontejneru (Docker exec zije nezavisle)
3. bbolt ma posledni stav "RUNNING"
4. crewshipd se restartne, zavola RecoverFromCrash()
5. Najde RUNNING runy v bbolt, zeptá se Dockeru jestli exec jeste bezi
6. Pokud ano: re-attach stdout, pokracuj
7. Pokud ne: oznac jako COMPLETED/FAILED

**Problem:** Behem crashu se ztratily WebSocket connections — browser nevidel zbytek outputu.
**Reseni:** Browser reconnect + "catch up" — po reconnectu precist JSONL log a zobrazit chybejici zpravy.

To je dobre navrzene v architecture.md (graceful shutdown). Ale AGENT-RUNTIME.md by mel explicitne popsat recovery flow pro JSONL catch-up.

### B7. Webhook secret rotation — uzivatel musi update vsechny integrace

Agent ma `webhook_secret`. Kdyz se rotuje, vsechny external systems (Grafana, n8n) musi dostat novy secret.

**Problem:** Zadna automaticka propagace. Uzivatel musi rucne update kazdy webhook v Grafane, n8n atd.

**Reseni pro MVP:** OK — manual rotation, uzivatel vidi secret v UI a prekopiruje ho. Phase 2: dual-secret window (stary i novy secret funguje po dobu X hodin).

## C. CHYBEJICI VECI (co nikde neni popsane)

### C1. Jak presne vypadá WebSocket handshake?

API.md rika "POST /api/v1/ws/token → short-lived JWT". Ale:
- Kdo generuje JWT? Next.js (pres NextAuth secret) nebo crewshipd (vlastni secret)?
- Ma crewshipd pristup k NEXTAUTH_SECRET pro validaci? Nebo se validuje pres IPC?
- Jaky je presny JWT payload? `{userId, orgId, teamId, exp}`?

**Doporuceni:** Next.js generuje WS token (stejny NEXTAUTH_SECRET), crewshipd validuje JWT lokalne (jose knihovna v Go). Payload: `{sub: userId, org: orgId, exp: now+5min}`. Team membership se overuje pres IPC az pri subscribe.

### C2. Jak se dostavaji agent eventy do browseru kdyz uzivatel NENI na strance?

Uzivatel posle webhook → agent bezi → uzivatel prijde na stranku pozdeji.

**Odpoved:** Agent output je v JSONL. Kdyz uzivatel otevre chat, Next.js/Go nacte JSONL a zobrazi historii. Real-time streaming je jen kdyz je uzivatel pripojeny pres WebSocket.

To je spravne navrzene — neni to problem.

### C3. Jak se resi file browser pro velke /output/ adresare?

Pokud agent vyproducuje 10,000 souboru, jak se zobrazuje file tree?
- Lazy loading (expand po adresarich)?
- Pagination?
- Search?

**Doporuceni:** Lazy loading per adresar (Go service vrati jen prvni uroven pri GET /teams/{id}/files, dalsi urovne na vyzvani). Search pres `filepath.Walk` v Go.

### C4. Jak se resi concurrent access k JSONL souboru?

Dva agenti ve stejnem tymu pisuji soucasne do logu. Go service pise do JSONL.

**Problem:** Concurrent file writes mohou zpusobit corrupted JSONL.

**Reseni:** Go service ma JEDINY writer goroutine per soubor (channel-based). Vsechny zapisy jdou pres kanal, writer je serializuje. To je spravne navrzene v Go architekture.

### C5. AGENTS.md rika "Prisma (ONLY DB access, from Next.js only)"

Ale co kdyz Go service potrebuje precist agent.webhook_secret pro webhook validaci? Pres IPC na Next.js — ok. Ale co kdyz Go service potrebuje updatovat AgentRun.status na COMPLETED?

**Odpoved:** Taky pres IPC. Go → Unix socket → Next.js → Prisma → DB.

To je dalsi roundtrip, ale drzi se principu "jedna cesta k DB". Prijatelne pro MVP.

## D. CO APLIKACE BUDE UMET (KOMPLETNI SEZNAM MVP)

### Pro uzivatele:
1. Registrace/login (email+heslo, Google, GitHub)
2. Vytvorit firmu (organizaci), pozvat kolegy
3. Vytvorit tymy (oddeleni), priradit lidi
4. Vytvorit AI agenty v tymech, pojmenovat je, dat jim roli ("DevOps inzenyr")
5. Priradit agentum skills (coding-assistant, web-researcher, customer-support)
6. Bezpecne ulozit API klice (Anthropic, OpenAI, atd.) a priradit je agentum
7. Chatovat s agentem v real-time (stream odpovedi)
8. Videt co agent dela live (thinking, tool calls, vysledky)
9. Prochazet a stahovat soubory ktere agent vytvoril (reporty, kod, data)
10. Nastavit webhooky — externi systemy mohou spustit agenta automaticky
11. Videt audit log — kdo co kdy udelal
12. RBAC — ruzne role s ruznymi opravnenimi
13. Dashboard — prehled tymu, agentu, jejich stavu

### Pro administratora:
14. Resource limity per tym (RAM, CPU, pocet agentu)
15. Billing plany (Free, Pro, Team, Enterprise) s limity
16. Feature flags (zapnout/vypnout funkce per organizace)
17. Smazani organizace s GDPR compliance (30d grace period)

### Pro externi systemy (webhooky):
18. Grafana alert → agent reaguje
19. n8n workflow → agent analyzuje
20. Jiny Crewship instance → agent-to-agent komunikace
21. CI/CD failure → agent analyzuje a fixne

### Technicky:
22. Real-time file notifikace (fsnotify) — "agent vytvoril novy soubor"
23. Container metriky (cAdvisor) — CPU, RAM per tym
24. Prometheus metriky — WebSocket connections, agent runs, errors
25. Web terminal — SSH-like pristup do kontejneru z browseru
26. Graceful shutdown — agent runs dokonci, logy se flusnou
27. Crash recovery — po restartu Go service se agent runs obnovi
